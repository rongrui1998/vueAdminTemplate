# V2 CRUD 页面生成 Skill 设计稿

## 1. 背景与目标

`front/V2` 的定位是一个可直接交付给同事复用的 Vue 后台模板。当前仓库已经具备：

- 登录、权限、动态菜单、动态路由基础链路。
- `SearchForm`、`ProTable`、`ModalForm`、`DrawerForm` 等业务组件。
- mock / API 双模式和 Node.js JSON 后端验证能力。
- 现成的标准业务模板页，可作为复制起点。

本需求希望把这些能力进一步抽象成一个可演示、可复用的 skill 流程，让 AI 能基于接口文档和自然语言，自动完成一个标准后台 CRUD 页面生成。

核心目标不是“让 AI 随便写一页代码”，而是形成一套可复制的业务交付流程：

1. 输入接口文档，可以是 Swagger URL，也可以是粘贴后的文本。
2. 输入自然语言业务描述，或者直接让 AI 从接口文档反推业务意图。
3. AI 自动识别业务对象、字段、列表、表单、详情、状态、操作按钮。
4. AI 自动生成页面代码、接口封装、静态兜底数据和菜单接入信息。
5. 页面默认优先请求真实接口，接口通或结构匹配则渲染接口数据。
6. 如果接口暂未准备好，页面使用同目录静态数据兜底，保证可演示。

这个能力特别适合模板型后台项目，因为同事拿到后可以直接复用同一套流程做新业务。

## 2. 设计原则

### 2.1 先理解，再生成

skill 的第一目标是把接口文档翻译成结构化业务规格，而不是直接输出代码。

原因是：

- 接口文档里通常只有字段和路径，没有完整页面语义。
- 生成前先确认业务理解，可以减少字段误判和页面返工。
- 先出规格更适合复用，也更容易让领导和同事理解。

### 2.2 页面优先复用项目既有组件

生成页面时优先使用当前模板已有的能力：

- 查询区优先 `SearchForm`
- 表格区优先 `ProTable`
- 新增和编辑优先 `ModalForm`
- 详情和预览优先 `DrawerForm`

如果业务特别复杂，再允许回退到原生 Element Plus 组件。

### 2.3 不额外引入单独 mock 体系

新生成的业务页面不需要单独的 `mock.ts`、`config.ts` 或全局 mock 接入规则。

页面只保留：

- 页面本体
- 接口封装
- 静态兜底数据
- 必要的子组件

这样更轻，也更适合“拿来即用”的模板目标。

### 2.4 路由由菜单驱动，不手改路由骨架

当前项目的业务页不是写死在 `router/index.ts` 里，而是由菜单数据生成动态路由。

因此这个 skill 在生成新页面时，必须同时给出菜单接入信息，确保页面能够被侧边栏和动态路由自动识别。

## 3. Skill 定位

建议 skill 名称：

- `CRUD 页面生成助手`

定位说明：

- 面向标准后台业务页生成
- 适合列表页、详情页、增删改查、状态切换
- 适合接口文档驱动的模板化开发
- 适合模板仓库对外演示和团队交付

## 4. 输入与输出

### 4.1 输入

支持以下输入形式：

- Swagger URL + 自然语言
- 接口文档文本 + 自然语言
- 仅接口文档，由 AI 自动补充业务理解后再确认

推荐演示时使用最后一种方式，因为更能体现“AI 先理解再生成”的价值。

### 4.2 输出

skill 先输出一份业务生成草案，再根据确认结果输出代码。

草案至少包含：

- 业务名称
- 页面标题
- 菜单名称
- 路由路径
- 组件路径
- 列表字段
- 查询字段
- 表单字段
- 详情字段
- 操作按钮
- 数据来源策略
- 文件清单

最终输出至少包含：

- 页面 `.vue`
- `api.ts`
- `data.ts`
- `types.ts`
- 必要子组件
- 菜单配置片段

## 5. 推荐执行步骤

### 步骤 1：解析接口文档

AI 从 Swagger 或文本中提取：

- 模块名称
- 列表接口
- 新增接口
- 编辑接口
- 删除接口
- 详情接口
- 分页参数
- 搜索条件
- 返回字段
- 状态字段
- 枚举值
- 时间字段
- 主键字段

### 步骤 2：推导业务规格

AI 把接口信息转成页面规格，例如：

- 页面标题：设备管理
- 列表字段：设备 ID、设备名称、设备类型、状态、创建时间、更新时间
- 查询字段：设备 ID、状态、创建时间范围
- 表单字段：设备名称、设备类型、状态、备注
- 行操作：编辑、删除、详情
- 是否需要状态切换：是
- 是否需要详情抽屉：是

### 步骤 3：生成草案并等待确认

AI 先输出草案，不直接写入代码。

使用者确认的内容包括：

- 页面标题是否准确
- 字段是否完整
- 哪些字段要隐藏
- 哪些字段要作为枚举
- 哪些按钮要保留
- 是否需要详情抽屉
- 菜单挂载位置是否正确

### 步骤 4：生成页面代码

确认后生成代码：

- 页面入口
- API 封装
- 静态兜底数据
- 类型定义
- 子组件

### 步骤 5：生成菜单接入信息

同时生成菜单配置建议，确保新页面能被动态路由识别并显示到侧边栏。

### 步骤 6：验证页面链路

最终验证：

- 页面可打开
- 查询可用
- 新增可用
- 编辑可用
- 删除可用
- 详情可用
- 接口通时走真实数据
- 接口不通时回退静态数据

## 6. 页面生成规则

### 6.1 目录建议

建议生成到独立业务目录，例如：

```text
front/V2/src/views/business/device-management/
  index.vue
  api.ts
  data.ts
  types.ts
  components/
    DeviceFormDialog.vue
    DeviceDetailDrawer.vue
```

### 6.2 文件职责

- `index.vue`
  - 页面入口
  - 请求并渲染列表
  - 组合现有业务组件

- `api.ts`
  - 真实接口封装
  - 后续联调时只改这里

- `data.ts`
  - 静态兜底数据
  - 用于接口未准备好时保证页面可展示

- `types.ts`
  - 接口字段类型和页面模型

- `components/*`
  - 业务弹窗、详情抽屉、可拆分复杂区域

### 6.3 数据策略

页面运行时遵循：

1. 先请求真实接口。
2. 成功并且结构符合预期，则直接渲染接口数据。
3. 失败或暂未接通接口，则使用同目录静态数据。

这套策略满足“接口通就直接用接口，不通也能展示”的要求。

## 7. 菜单与路由规则

### 7.1 新页面是否需要改路由文件

通常不需要。

当前项目业务页的接入依赖菜单数据和动态路由生成流程，不是手工把业务页写进 `router/index.ts`。

### 7.2 新页面如何显示

新页面需要同步新增菜单数据，菜单中至少要包含：

- `name`
- `path`
- `component`
- `type`
- `sort`
- `permission`
- `status`

其中 `component` 必须能映射到真实视图文件路径。

### 7.3 component 字段规则

菜单里的 `component` 会映射到：

```text
src/views/${component}.vue
```

例如：

```ts
component: 'business/device-management/index';
```

对应：

```text
front/V2/src/views/business/device-management/index.vue
```

如果路径写错，路由会进入 `route-missing`，用于提示动态路由配置异常。

## 8. 建议的 skill 输出格式

建议 skill 在确认后输出一个结构化结果，便于后续自动化或人工复制。

示例：

```json
{
  "moduleName": "设备管理",
  "pageTitle": "设备管理",
  "menuName": "设备管理",
  "routePath": "/business/device-management",
  "component": "business/device-management/index",
  "dataStrategy": "interface-first-with-static-fallback",
  "searchFields": ["deviceId", "status", "createdTimeRange"],
  "tableColumns": ["deviceId", "deviceName", "deviceType", "status", "createdAt", "updatedAt"],
  "formFields": ["deviceName", "deviceType", "status", "remark"],
  "actions": ["add", "edit", "delete", "detail"],
  "files": [
    "front/V2/src/views/business/device-management/index.vue",
    "front/V2/src/views/business/device-management/api.ts",
    "front/V2/src/views/business/device-management/data.ts",
    "front/V2/src/views/business/device-management/types.ts"
  ]
}
```

## 9. 给领导演示时的讲法

演示重点不只是“AI 能写页面”，而是“AI 能把接口文档变成可交付页面流程”。

建议演示链路：

1. 输入 Swagger URL 或接口文档。
2. 输入一句简短需求，例如“生成设备管理 CRUD 页面，优先使用项目现成组件”。
3. AI 自动输出业务理解草案。
4. 使用者确认字段和页面结构。
5. AI 生成页面、接口封装、静态兜底数据和菜单建议。
6. 页面接入菜单后自动出现在侧边栏。
7. 接口可用时渲染接口数据，接口未准备好时也能通过静态数据展示。

对外表达时可以强调：

- 不是单次写页面，而是标准化业务页生成流程
- 不是写死一个 demo，而是适合复用到任何 CRUD 业务
- 不是依赖复杂 mock 体系，而是页面自带静态兜底，接口好后自然切换
- 不是手动改路由，而是菜单驱动自动接入

## 10. 适用范围与边界

### 适用范围

- 标准列表页
- 增删改查
- 分页和筛选
- 状态切换
- 详情抽屉
- 弹窗表单
- 业务字段较规整的后台场景

### 不建议第一版覆盖

- 特别复杂的多表联动
- 大型流程编排
- 图表分析页
- 复杂导入导出编排
- 低代码拖拽式页面搭建

原因是第一版目标是让 skill 稳定、可演示、可复用，而不是一次性覆盖全部业务形态。

## 11. 建议的后续落地顺序

1. 先把这个设计稿作为正式说明文档。
2. 再根据这个设计稿写 skill 的固定步骤和提示词结构。
3. 最后挑一个真实接口文档做演示样板，例如设备管理。
4. 演示时优先走“文档 -> 草案 -> 确认 -> 生成 -> 菜单接入 -> 页面展示”的完整链路。
