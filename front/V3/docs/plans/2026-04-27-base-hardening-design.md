# V3 Base 补强设计

## 背景

当前 `front/V3` 的基础链路已经完整，能够完成登录、动态菜单、动态路由、标签页、mock CRUD 和基础布局演示。但相比 `V3Lite`，仍然存在 4 类明显短板：

- 权限颗粒度停留在页面级兜底，缺少按钮级权限能力
- 无权限访问仍然回退到 `404`，语义不够准确
- 缺少自动化测试和提交前质量护栏
- 缺少统一入口文档，新人上手成本偏高

本次补强不改变 Base 的定位，不引入 RBAC、数据权限或复杂后台配置页，只补“更稳、更可复用”的最小必要能力。

## 目标

本次补强后的 `V3 Base` 需要满足：

- 页面权限和按钮权限可以共用一套 permission 语义
- 未授权访问有独立 `403` 页面
- 路由守卫、菜单转路由、请求层和 mock 契约有基础测试保护
- 项目具备 `stylelint`、`vitest`、`husky`、`lint-staged` 基础工程护栏
- 提供一个统一 `README` 作为对外入口

## 方案

### 1. 权限能力补强

新增两层能力：

- `v-permission` 指令：用于按钮和局部区块显隐
- `usePermission()` composable：提供 `hasPermission / hasAnyPermission / hasAllPermissions`

两者都以现有 `authStore.hasPermission()` 为唯一判定来源，避免再造一套权限状态。这样能保证菜单、页面和按钮权限口径一致，同时不扩大 Base 复杂度。

### 2. 403 页面与守卫语义修正

新增独立 `403` 页面，并在路由常量和静态路由中注册。

调整守卫逻辑：

- 未登录仍跳登录页
- 已登录访问登录页仍跳首页
- 页面缺少权限时从当前 `404` 调整为 `403`
- 保留 `route-missing` 和真正未知路由的 `404` 逻辑

这样可以把“菜单配置错误”和“用户无权限”这两类问题清晰分开。

### 3. 测试与质量工具

引入：

- `Vitest + @vue/test-utils + happy-dom`
- `Stylelint`
- `husky + lint-staged`

测试优先覆盖最容易回归的边界：

- 菜单转路由
- 路由守卫的登录态与权限分支
- 请求层 401 与报错行为
- mock 登录 / 用户 / 菜单契约

这次不追求测试面铺满，而是先把基础链路最关键的断点接住。

### 4. README 收口

新增 `front/V3/README.md`，集中说明：

- 项目定位
- 启动与校验命令
- 环境变量
- 登录与权限链路
- 如何新增页面与权限
- 当前 Base 范围与后续演进方向

已有 `doc/` 目录继续保留，README 负责“入口”和“导航”。

## 非目标

本次不包含：

- 用户/角色/菜单管理后台
- 数据权限
- 完整 RBAC 模型
- 国际化
- 大规模页面重构

## 验证策略

完成后至少验证：

- `pnpm lint`
- `pnpm stylelint`
- `pnpm test`
- `pnpm build`

如果安装新依赖导致锁文件变动，也一并纳入结果说明。
